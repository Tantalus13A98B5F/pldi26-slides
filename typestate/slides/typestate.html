Track state changes like types — statically
So first: what is typestate? In simple words, typestate means tracking the *state changes* of objects the same way we track types — statically, at compile time.

Example (1): Database Locks

Tracking lock states = flexible imperative code + static safety

Scoped Blocks: nested lock lifetimes❓ 

        synchronized (table) {
            var row = table.locateRow(...);
            synchronized (row) {
                return computeOnRow(row);
            }
        }

Hand-over-Hand: better parallelism ✅

The classic example is locking. Languages like Java offer implicit lock management tied to lexical scopes. That's simple to use, but lock lifetimes are then forced to nest, so you cannot release the table while still working on a row.
In contrast, by using Lock and Unlock methods directly, imperative code allows us to unlock the table *as soon as* we lock the row — that's hand-over-hand locking, which offers better parallelism.
By typestate tracking on locks, we can further statically ensure that we never use resources without locking them.

Example (2): Well-formed DOM Trees

Tracking lists of open tags = context-free grammar (CFG) 

        makeDOM { dom =>
          dom.open(DIV())         // dom: [] -> [DIV]
          // adding text... 

          // !!Error:!! [DIV] got, [..., P] expected
          dom.close(P())

          // Ok
          dom.close(DIV())        // dom: [DIV] -> []

          // !!Error:!! [] got, [..., DIV] expected
          dom.close(DIV())
        }
The next example is about building DOM trees; a tree is well-formed only when all tags are properly nested. The state we track is no longer a simple flag, but the entire stack of currently-open tags. Here, we demonstrate the checker catching two possible errors: closing a tag that doesn't match, and closing when no tag is open at all. The key bit here is that: typestate is not limited to a finite set of named states; it can track structured states like a stack, and use it to enforce grammar-shaped invariants, entirely at compile time.

Example (3): Server/Client Communication

Tracking channel protocols throughout recursion, branching... 

      // server: prints text from client until client quits
      def echoServer(chan: Chan): ... =
          chan.RecPush()                   // chan: Rec[Recv[…]]           -> Recv[String,…]

          def recur(chan: Chan): ... = 
              println(chan.Recv())         // chan: Recv[String,Branch[…]] -> Branch[Var[0],End]
              chan.Branch {
                  chan.RecTop()            // chan: Var[0]                 -> Recv[String, …]
                  recur(chan)              // chan: Recv[String, …]        -> End
              } {
                  chan.Close()             // chan: End
              }

          recur(chan)                      // chan: Recv[String, …]        -> End
      
      // client: repeatedly sends text until quitting
      def echoClient(chan: Chan) = ...     // (dual of echoServer)
Our last example is about communication protocols, or session types. The two ends of a channel must follow a protocol, a predefined sequence of sends and receives, and crucially, that protocol keeps evolving as the program progresses. In our example, the server keeps receiving strings until the client decides to quit, and only then closes the channel. Don't worry about the exact state transitions here; but the point is that we must track the channel's state statically even through recursion and branching.
Revocable capabilities for general typestate

Typestate in more general settings?

As our running example, you may only read or write a file while it's open. There are really two ways to write such code. First, the direct style — you just call Open, Write, and Close as you go. To make that safe, you need full typestate, and people have explored this for a long time, from early whole-program analyses, to typestate-oriented languages like Plaid, and session types. But the existing solutions tend to come at a price — they're tied to specialized languages or specialized domains, or they impose restrictions on aliasing, using mechanisms like linear types and fractional permissions.

Alternatively, in the scoped style — you wrap the file in a WithFile block, and now the only thing you need for safety is that the capability f doesn't escape that scope. That's a much lighter requirement, and several proposals have been made recently — second-class values, Reachability Types, and Scala's Capturing Checking — the last of which is already running inside a mainstream language. So the natural question is: can this lightweight, capability-based path be generalized to express full typestate?

Our answer is yes — without inventing a whole new type system for Scala, we can describe stateful methods like Open using a single signature, where that funny arrow in the middle explains the process. Reading it left to right, it does three things: it receives the closed-state capability, revokes it to stop further use, and returns the new open-state capability. Our receiving simply reuses Scala's implicits; revoking requires an effect system that builds on Reachability Types and Capturing Checking; returning a capability is our new extensions and is implemented via a simple transformation.

The Very First Step

Use different types for different states: 

    class OpenFile: ...
    class ClosedFile: ...

    def NewFile(name: String):  ClosedFile = ...
    def Open   (f: ClosedFile): OpenFile   = ...    //: ClosedFile -> OpenFile 
    def Close  (f: OpenFile):   ClosedFile = ...    //: OpenFile -> ClosedFile

    def Read (f: OpenFile): String         = ...
    def Write(f: OpenFile, text: String): Unit = ...
Benefit: Write only happens on OpenFile 

    val fNew = NewFile("a.txt")
    val fOpen = Open(fNew)
    Write(fOpen, "Hello")                           // Good: on OpenFile
    val fClosed = Close(fOpen)
To see why this setup is necessary, let's build it up naively first. The simplest encoding is just to use different types for different states: an OpenFile and a ClosedFile as distinct classes. A NewFile is always Closed; Open converts that to an OpenFile; and Close goes the other way. Crucially, Read and Write require an OpenFile. So already, the type checker guarantees you can only Write to an OpenFile.

This naive version already gets us a lot, but it has three serious problems. First, after Close gives you a fresh ClosedFile, the old fOpen handle is stale — but nothing stops you from writing to it. Second, if you open file A, the resulting OpenFile doesn't actually remember it's A, so you might accidentally write content meant for B into A. Third, because every operation returns a new handle, you're forced to thread a chain of freshly-named variables through your whole program. These three pain points motivate our three pillars: flow-sensitive revocation kills stale handles, path-dependent capabilities tie a capability to its object, and flow-sensitive implicits remove the naming burden. We detail these pillars in the next couple of slides.

Pillar 1: Flow-Sensitive Revocation

Ensure outdated capabilities (by @kill) are no longer used: 

    def Close(f: OpenFile^): ClosedFile^ @kill(f) = ...  //: OpenFile^ =!> ClosedFile^

    val fClosed = Close(fOpen)    // ??{ fOpen }?? marked as killed
    Write(fOpen, "Hello")         // !!Error:!! using killed fOpen
Key: disjoint use & kill under descriptive alias tracking
  • track sharing by names, pose minimum restrictions
  • Reachability Types (RT): destructive effects proven sound [1]
  • Capturing Types: experimental checker (CC) in Scala compiler

      val fA = Open(NewFile("a.txt"))
      val fB = Open(NewFile("b.txt"))
      val fC = if (...) fA else fB    // fC: OpenFile??~{fA,fB}??
      Close(fA)
      Write(fC, "maybe to a.txt")     // !!Error:!! using killed fA

Pillar one: flow-sensitive revocation. We annotate Close with a kill effect on f, meaning the argument f is consumed. After calling Close fOpen, any later use of fOpen is flagged as using a killed capability — that fixes the stale-handle problem.

The interesting part is doing this under aliasing. We use what we call descriptive alias tracking: we track sharing by names, so we impose only the minimum restriction. Such mechanisms include Reachability Types and Capturing Types, where destructive effects are proven sound in the context of RT, and we build our revocation based on the experimental CT checker in Scala. The example below demonstrates such tracking: fC may alias either fA or fB. When we Close fA, the checker knows fC could reach the killed fA, so writing through fC is correctly rejected.

Pillar 2: Path-Dependent Capabilities

Disentangle capabilities as abstract type members of objects: 

    class File:
        type IsClosed   // abstract type members
        type IsOpen
Use path-dependent types (DOT) to relate the capability c to the object f: 

    def Open(f: File, c: f.IsClosed^): f.IsOpen^ @kill(c) = ...

    Open(f, c)        // separate obj and cap, kill only the cap

unaffected by kill effects;   guaranteed relation to objects

Pillar two: path-dependent capabilities. Instead of using different types for different states, we make capabilities abstract type members of File objects: every File has its own IsClosed and IsOpen types. Then, using path-dependent types from DOT — the calculus underlying Scala — a capability of type f.IsClosed is provably about the file f and no other. So Open takes the object f and separately the IsClosed capability c specific to f, and then kills only the capability, not the object. This path-dependency is immune to kill effects from pillar one, and it gives us a guaranteed link between a capability and the object it governs — so a write does not silently go to the wrong file.

Pillar 3: Flow-Sensitive Implicits

Existing Scala feature for receiving capabilities, implicitly: 

    def Open(f: File)(using c: f.IsClosed^) = ...  // f.IsClosed^ ?=!> f.IsOpen^

    Open(f)                                        // implicitly using nearest f.IsClosed
New extension: returning capabilities implicitly via ANF transformation 

    // bundle data & capability as dependent pair: Sigma
    def NewFile(name: String): Sigma { type A = File; type B = a.IsClosed^ } = ...

    val f = NewFile("...")   // non-tail Sigma ==> (1) .a as f; (2) .b as implicit cap
    Open(f)                  //                   (3) continue on
Pillar three: flow-sensitive implicits — this improves ergonomics. Half of it is already in Scala: instead of passing the capability by hand, we mark it with "using" so the compiler finds the nearest implicit candidate, and you just write Open f. The new half that makes implicits really flow-sensitive is their returning. The trick here is that: when a function returns both the data and a capability as a dependent pair, or Sigma, our ANF transformation kicks in and extracts the second component as an implicit candidate for the rest of the block. So after assigning the result of NewFile to f, the IsClosed capability is silently available, and the next Open just picks it up. No more threading chains of renamed variables — the program reads like ordinary imperative code.

Control Flow, Performance
Now the benefits — what this buys us in real control flow, and what it costs.

Control Flow: Non-forking Usage

Using capabilities need no special handling at all: 

    val f = NewFile("a.txt")
    Open(f)
    for (ln <- text)
        Write(f, ln)   // using f.IsOpen in a loop
    Close(f)
Ordinary if for branching that lands in the same state 

  // from echoClient.recur: chan.PCap[...] ?=!> Unit
  if (readLine() == …) then
      chan.Left(); chan.RecTop()
      recur(chan)
  else
      chan.Right(); chan.Close()
First, the easy case is when control flow *doesn't* fork the ending state. Using a capability repeatedly needs no special treatment at all — here we Write to f inside an ordinary for-loop, where each iteration leaves the state unchanged. Likewise, a plain if-then-else is fine as long as both branches land in the *same* state. In this session-types snippet, both the `then` and `else` branches fully execute the protocol and close the channel, so we just use a normal if. The takeaway is: a lot of real code needs zero extra work.

Control Flow: Forking & Joining States

Forking states into Either: 

      //: f.IsClosed ?=!>?
      //  Either[f.IsOpen, f.IsClosed]
      if_[..., ..., ...] (...)
      /* then */ {
          Open(f)
      } /* else */ {
          Open(f); Close(f)
      }
Joining by inspecting Either values: 

      //: Either[f.IsOpen, f.IsClosed] ?=!>?
      //  f.IsClosed
      match_[..., ..., ...]
      { // case f.IsOpen =>
          println("Now closing."); Close(f)
      }{// case f.IsClosed =>
          println("Already closed.")
      }
Combining them: loop guarded by state transition 

    // iterate it until no more element
    while_[it.HasMore, it.End] {
        //: it.HasMore ?=!> (Either[it.HasMore, it.End] ?<= T)
        val item = it.nextChecked()
        ...
    }

The harder case is when branches end in *different* states. We handle this by reifying the choice as Either capabilities, and we offer combinators to work with them. Our if-combinator runs one branch that leaves the file open and another that leaves it closed, and its result is Either of IsOpen and IsClosed — the state becomes data. Joining is the dual: a match-combinator inspects that Either, and in each case the capability is recovered with the right state, so both cases can converge — here, back to closed.

Combining fork and join gives us state-dependent loops: the while-combinator keeps iterating until the ending state is reached. So even non-trivial control flow stays expressible and checked.

To wrap up. We bring typestate to Scala — and to general settings — by treating stateful programming as *receiving, revoking, and returning* capabilities. We get ergonomic control flow and impose only minimal aliasing restrictions. Our three pillars include: destructive effects for revocation, path-dependent capabilities for tying state to objects, and — the new ingredient — implicit return values via ANF. We validated it on files, locks, DOM trees, session types, and control flow combinators, and our artifact is available online. Thanks very much — I'm happy to take questions.
